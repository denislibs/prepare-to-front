# Назовите средства и методы тестирования доступности?

Тестирование доступности — это процесс проверки веб-сайтов и приложений на соответствие стандартам доступности (WCAG) и обеспечение того, что они могут использоваться людьми с различными ограниченными возможностями.

## Автоматические инструменты тестирования

### 1. **WAVE (Web Accessibility Evaluation Tool)**

Бесплатный инструмент от WebAIM для оценки доступности.

**Особенности:**
- Расширение для браузеров (Chrome, Firefox, Edge)
- Онлайн-версия
- API для автоматизации
- Визуальная индикация проблем

**Использование:**
```bash
# Установка расширения в браузер
# Затем открыть страницу и нажать на иконку WAVE
```

### 2. **axe DevTools**

Мощный инструмент для автоматического тестирования доступности.

**Особенности:**
- Расширение для браузеров
- Интеграция с CI/CD
- Подробные отчеты
- Поддержка WCAG 2.1

**Использование:**
```javascript
// Программное использование
import axe from 'axe-core';

axe.run(document, (err, results) => {
  if (err) throw err;
  console.log(results);
});
```

### 3. **Lighthouse**

Встроенный инструмент Chrome DevTools для аудита доступности.

**Особенности:**
- Встроен в Chrome DevTools
- Командная строка
- CI/CD интеграция
- Оценка от 0 до 100

**Использование:**
```bash
# В Chrome DevTools
# F12 → Lighthouse → Accessibility

# Командная строка
lighthouse https://example.com --only-categories=accessibility
```

### 4. **Pa11y**

Командная строка для автоматического тестирования.

**Особенности:**
- CLI инструмент
- CI/CD интеграция
- Поддержка различных стандартов
- JSON/HTML отчеты

**Использование:**
```bash
# Установка
npm install -g pa11y

# Тестирование
pa11y https://example.com

# С конфигурацией
pa11y --standard WCAG2AA https://example.com
```

### 5. **Accessibility Insights**

Инструмент от Microsoft для тестирования доступности.

**Особенности:**
- Расширение для браузеров
- Windows приложение
- Подробные рекомендации
- Интеграция с Azure DevOps

## Ручное тестирование

### 1. **Тестирование с клавиатуры**

```html
<!-- ✅ Проверка навигации с клавиатуры -->
<!-- Tab - переход вперед -->
<!-- Shift+Tab - переход назад -->
<!-- Enter/Space - активация -->
<!-- Стрелки - навигация в меню/списках -->
```

**Что проверять:**
- Все интерактивные элементы доступны с клавиатуры
- Видимый индикатор фокуса
- Логический порядок навигации
- Нет "ловушек" фокуса

### 2. **Тестирование со скринридерами**

#### NVDA (Windows)
```bash
# Бесплатный скринридер
# Скачать с nvaccess.org
# Использовать для навигации по странице
```

**Что проверять:**
- Структура заголовков (H1-H6)
- Альтернативный текст изображений
- ARIA атрибуты работают правильно
- Формы имеют метки
- Навигация понятна

#### VoiceOver (macOS/iOS)
```bash
# Встроенный скринридер
# Cmd+F5 для включения
# Использовать для навигации
```

#### JAWS (Windows)
```bash
# Коммерческий скринридер
# Самый популярный
# Использовать для тестирования
```

### 3. **Проверка контрастности**

#### Color Contrast Analyzer
```bash
# Инструмент для проверки контрастности
# Доступен как расширение браузера
# Проверяет WCAG AA и AAA уровни
```

**Что проверять:**
- Контраст текста и фона (4.5:1 для AA, 7:1 для AAA)
- Контраст крупного текста (3:1 для AA, 4.5:1 для AAA)
- Контраст интерактивных элементов

### 4. **Проверка масштабирования**

```css
/* ✅ Проверка увеличения текста до 200% */
/* Браузер: Ctrl/Cmd + Plus */
/* Должно работать без горизонтальной прокрутки */
```

**Что проверять:**
- Текст увеличивается до 200%
- Нет потери функциональности
- Нет горизонтальной прокрутки
- Контент остается читаемым

## Методы тестирования

### 1. **Автоматическое тестирование**

```javascript
// Пример с Jest и axe-core
import { axe, toHaveNoViolations } from 'jest-axe';

expect.extend(toHaveNoViolations);

test('should have no accessibility violations', async () => {
  const { container } = render(<MyComponent />);
  const results = await axe(container);
  expect(results).toHaveNoViolations();
});
```

### 2. **Тестирование с пользователями**

**Методы:**
- Интервью с пользователями с ограниченными возможностями
- Наблюдение за использованием
- Сбор обратной связи
- A/B тестирование доступных версий

### 3. **Эвристическая оценка**

**Чек-лист:**
- [ ] Все изображения имеют alt текст
- [ ] Формы имеют метки
- [ ] Заголовки используются правильно
- [ ] Контраст достаточен
- [ ] Навигация с клавиатуры работает
- [ ] ARIA атрибуты используются правильно

## CI/CD интеграция

### GitHub Actions

```yaml
# .github/workflows/accessibility.yml
name: Accessibility Test

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Run Pa11y
        run: |
          npm install -g pa11y
          pa11y https://example.com
```

### Пример отчета

```json
{
  "issues": [
    {
      "type": "error",
      "code": "WCAG2AA.Principle1.Guideline1_4.1_4_3.G18.Fail",
      "message": "Контраст текста недостаточен",
      "element": ".header-text"
    }
  ]
}
```

## Лучшие практики

### ✅ Делайте:

1. **Комбинируйте автоматическое и ручное тестирование**
2. **Тестируйте на ранних этапах разработки**
3. **Используйте несколько инструментов**
4. **Тестируйте с реальными пользователями**
5. **Документируйте результаты**

### ❌ Не делайте:

1. **Не полагайтесь только на автоматическое тестирование**
2. **Не игнорируйте ручное тестирование**
3. **Не тестируйте только в одном браузере**
4. **Не забывайте тестировать с клавиатуры**
5. **Не пропускайте тестирование со скринридерами**

## Заключение

Тестирование доступности включает:

- **Автоматические инструменты** — WAVE, axe, Lighthouse, Pa11y
- **Ручное тестирование** — клавиатура, скринридеры, контрастность
- **Тестирование с пользователями** — реальный опыт использования
- **CI/CD интеграция** — автоматизация в процессе разработки

**Помните:** автоматическое тестирование находит только около 30% проблем доступности. Всегда комбинируйте автоматическое и ручное тестирование, включая тестирование с реальными пользователями и вспомогательными технологиями.

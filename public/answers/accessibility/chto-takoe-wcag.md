# Что такое WCAG?

WCAG (Web Content Accessibility Guidelines) — это набор рекомендаций по обеспечению доступности веб-контента, разработанный консорциумом W3C (World Wide Web Consortium). Эти руководящие принципы помогают сделать веб-сайты и веб-приложения доступными для людей с различными ограничениями, включая нарушения зрения, слуха, моторики и когнитивные расстройства.

## Основная информация

WCAG был впервые опубликован в 1999 году (WCAG 1.0), затем обновлен в 2008 году (WCAG 2.0) и в 2018 году (WCAG 2.1). Последняя версия — WCAG 2.2, опубликованная в 2023 году.

## Структура WCAG

WCAG основан на четырех основных принципах доступности, которые часто называют аббревиатурой **POUR**:

### 1. **Perceivable (Воспринимаемость)**
Информация и компоненты интерфейса должны быть представлены пользователям таким образом, чтобы они могли их воспринимать.

**Примеры:**
- Предоставление текстовых альтернатив для изображений
- Использование достаточного контраста между текстом и фоном
- Возможность изменения размера текста без потери функциональности

```html
<!-- Хороший пример -->
<img src="chart.png" alt="График продаж за 2023 год: рост на 25%">

<!-- Плохой пример -->
<img src="chart.png">
```

### 2. **Operable (Управляемость)**
Компоненты интерфейса и навигация должны быть управляемыми.

**Примеры:**
- Все функции доступны с клавиатуры
- Достаточное время для чтения и использования контента
- Контент не вызывает судорог или физических реакций

```html
<!-- Хороший пример - кнопка доступна с клавиатуры -->
<button onclick="submitForm()" tabindex="0">Отправить</button>

<!-- Плохой пример - div с обработчиком клика -->
<div onclick="submitForm()">Отправить</div>
```

### 3. **Understandable (Понятность)**
Информация и работа пользовательского интерфейса должны быть понятными.

**Примеры:**
- Читаемый и понятный текст
- Предсказуемая работа веб-страниц
- Помощь пользователям в избежании и исправлении ошибок

```html
<!-- Хороший пример - понятное сообщение об ошибке -->
<label for="email">Email</label>
<input type="email" id="email" aria-describedby="email-error">
<span id="email-error" role="alert">Пожалуйста, введите корректный email адрес</span>

<!-- Плохой пример - непонятное сообщение -->
<input type="email" id="email">
<span>Ошибка 404</span>
```

### 4. **Robust (Надежность)**
Контент должен быть достаточно надежным для интерпретации различными пользовательскими агентами, включая вспомогательные технологии.

**Примеры:**
- Валидная разметка HTML
- Правильное использование ARIA атрибутов
- Совместимость с текущими и будущими технологиями

```html
<!-- Хороший пример - валидная разметка -->
<nav aria-label="Основная навигация">
  <ul>
    <li><a href="/">Главная</a></li>
    <li><a href="/about">О нас</a></li>
  </ul>
</nav>
```

## Уровни соответствия

WCAG определяет три уровня соответствия:

### **Уровень A (Минимальный)**
Базовый уровень доступности. Веб-сайт должен соответствовать всем критериям уровня A.

**Примеры требований:**
- Все изображения имеют текстовые альтернативы
- Заголовки используются для структурирования контента
- Формы имеют метки

### **Уровень AA (Рекомендуемый)**
Средний уровень доступности. Большинство организаций стремятся к этому уровню.

**Примеры требований:**
- Минимальный коэффициент контрастности 4.5:1 для обычного текста
- Масштабирование текста до 200% без потери функциональности
- Навигация с клавиатуры доступна для всех интерактивных элементов

### **Уровень AAA (Максимальный)**
Высший уровень доступности. Достичь этого уровня для всего контента может быть сложно.

**Примеры требований:**
- Коэффициент контрастности 7:1 для обычного текста
- Язык страницы и изменения языка указаны в коде
- Контекстная справка доступна

## Практические примеры применения WCAG

### Пример 1: Доступная форма

```html
<form>
  <fieldset>
    <legend>Контактная информация</legend>
    
    <label for="name">Имя <span aria-label="обязательное поле">*</span></label>
    <input 
      type="text" 
      id="name" 
      name="name" 
      required
      aria-required="true"
      aria-describedby="name-help"
    >
    <span id="name-help" class="help-text">
      Введите ваше полное имя
    </span>
    
    <label for="email">Email <span aria-label="обязательное поле">*</span></label>
    <input 
      type="email" 
      id="email" 
      name="email" 
      required
      aria-required="true"
      aria-invalid="false"
      aria-describedby="email-error"
    >
    <span id="email-error" role="alert" aria-live="polite"></span>
  </fieldset>
  
  <button type="submit">Отправить</button>
</form>
```

### Пример 2: Доступная навигация

```html
<nav aria-label="Основная навигация">
  <ul>
    <li><a href="/" aria-current="page">Главная</a></li>
    <li><a href="/products">Продукты</a></li>
    <li><a href="/about">О нас</a></li>
    <li><a href="/contact">Контакты</a></li>
  </ul>
</nav>

<!-- Skip links для улучшения навигации -->
<a href="#main-content" class="skip-link">
  Перейти к основному контенту
</a>
```

### Пример 3: Доступные медиа-элементы

```html
<!-- Видео с субтитрами и описанием -->
<video controls>
  <source src="video.mp4" type="video/mp4">
  <track 
    kind="captions" 
    src="captions.vtt" 
    srclang="ru" 
    label="Русские субтитры"
    default
  >
  <track 
    kind="descriptions" 
    src="descriptions.vtt" 
    srclang="ru"
  >
  Ваш браузер не поддерживает видео.
</video>

<!-- Аудио с транскриптом -->
<audio controls>
  <source src="audio.mp3" type="audio/mpeg">
</audio>
<a href="transcript.txt">Транскрипт аудио</a>
```

## Инструменты для проверки соответствия WCAG

1. **WAVE** (Web Accessibility Evaluation Tool) — расширение для браузера
2. **axe DevTools** — инструмент для автоматического тестирования
3. **Lighthouse** — встроенный инструмент Chrome DevTools
4. **NVDA/JAWS** — программы чтения с экрана для ручного тестирования
5. **Color Contrast Analyzer** — проверка контрастности цветов

## Важность WCAG

Следование WCAG важно по нескольким причинам:

1. **Юридические требования** — во многих странах есть законы, требующие доступности веб-сайтов
2. **Расширение аудитории** — доступные сайты могут использовать больше людей
3. **SEO преимущества** — многие практики доступности улучшают SEO
4. **Лучший UX** — доступные сайты обычно имеют лучший пользовательский опыт для всех
5. **Социальная ответственность** — обеспечение равного доступа к информации

## Заключение

WCAG — это не просто набор правил, а философия создания инклюзивного веб-пространства. Следование этим рекомендациям помогает создавать веб-сайты, которые могут использовать все люди, независимо от их способностей или ограничений.
